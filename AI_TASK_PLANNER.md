## User Input

- **Required target flight distance** (e.g. 100 km, 200 km) — with ±5% tolerance
- **Takeoff airport** — by ICAO/IATA code or name
- **Destination airport** — by ICAO/IATA code or name (may be same as takeoff for out-and-return or triangle tasks)
- **Glider** — selected from glider library; used to calculate estimated flight time and thermal performance
- **Maximum flight duration** — combined with glider performance, caps the viable task area
- **Flight date** — weather must be fetched for a specific day, not the whole year
- **Planned takeoff time** — if not given, AI suggests the best takeoff window based on thermal forecast
- **Pilot safety profile** — conservative / standard / aggressive (affects airspace buffer, minimum altitude margin, and weather thresholds)
- **Soaring mode** — thermal / ridge / wave (affects weather parameters fetched, scoring weights, altitude ceilings, and route geometry)
- **Airspace constraints** (granular toggles):
  - Include/exclude zones requiring a transponder
  - Include/exclude zones requiring a flight plan
  - Include/exclude controlled airspace classes (C, D, E)
  - Include/exclude restricted, danger, and prohibited zones

## Soaring Mode Considerations

The planner must account for the three fundamentally different soaring techniques. The selected soaring mode drives which weather parameters matter most, how routes are shaped, and how the scoring function is weighted.

### Thermal Soaring
- Primary lift source: convective thermals driven by surface heating
- Key weather inputs: BL height, thermal strength (CAPE, Lifted Index, CIN), surface temperature, cloud base, Cu development
- Route logic: prefer grid cells with BL ≥ 1200 m and positive thermal indices; avoid overdeveloped cells (Cb risk); plan tasks for the midday–early afternoon thermal window
- Altitude ceiling: BL top (convective mixing layer); cloud base if Cu present
- Terrain role: identify south-facing slopes and dark surface areas as thermal trigger hotspots; avoid large water bodies (thermal deserts)
- Scoring weight: thermal coverage is the dominant factor

### Ridge Soaring
- Primary lift source: orographic updraft on the windward side of ridges and escarpments
- Key weather inputs: wind speed and direction at ridge height (850–700 hPa), consistency of wind (low directional variance), ridge orientation vs. wind angle
- Route logic: align task legs parallel to ridges where wind angle is 60–90° to the ridge axis; optimal lift band is 1–2× ridge height above the crest; plan out-and-return or linear tasks along the ridge
- Terrain role: critical — requires DEM data to identify viable ridge lines, windward slopes, and gap/leeside turbulence zones; route must stay on the windward side at all times
- Altitude ceiling: ridge height + estimated updraft depth (typically 300–600 m above crest)
- Scoring weight: wind alignment with terrain features is the dominant factor; thermal scoring is secondary

### Wave Soaring
- Primary lift source: mountain wave (standing wave) generated by stable airflow over terrain
- Key weather inputs: wind speed ≥ 25 kt at ridge height, stable lapse rate above the inversion, wave-favourable wind direction (within ~30° perpendicular to the mountain range), presence of lenticular cloud or clear-air wave (CAT indicators)
- Route logic: task legs run parallel to the wave rotor/bar system; wave windows are often narrow lateral bands — tasks are typically linear or elongated along the bar; avoid rotor zone below the wave (turbulent, lower half of wave cycle)
- Terrain role: identify the triggering mountain range and estimate wave wavelength (~10–20 km typical); tasks often start well downwind of the mountains
- Altitude ceiling: tropopause (wave can extend to FL200–FL600 depending on conditions); airspace above FL100/FL195 requires coordination — flag any wave bars that enter Class A or higher
- Scoring weight: wave bar alignment and stability index dominate; thermal scoring is not applicable

### Mixed Conditions
- If the weather analysis detects conditions suitable for more than one soaring type (e.g., morning ridge + afternoon thermal), note this in the task proposal and allow the pilot to choose or let the AI blend the scoring weights proportionally
- Always flag when wave conditions are present regardless of the selected mode — wave flying requires specific pilot qualification and often restricted airspace coordination

## Weather Analysis

### Area Coverage

- If takeoff == destination: use a circular mesh centered on the airport, radius = task distance + 20% buffer
- If takeoff ≠ destination: use a corridor mesh ±40 km either side of the straight line between the two airports — this is more efficient than a full circle
- In both cases use ~25 km spacing between grid points

### Time Window

- Do not analyze the whole day indiscriminately
- Analyze from 2 hours before planned takeoff through planned takeoff + maximum flight duration + 1 hour buffer
- This alone significantly reduces the number of weather data points needed

### Weather Parameters to Fetch

General:
- Wind speed and direction at 10 m, 850 hPa, 700 hPa, 500 hPa
- Visibility
- Surface temperature and dew point
- Cloud base height
- Cloud type (cumulus, cumulonimbus, stratus, cirrus)
- Precipitation probability and amount
- Pressure gradient and characteristics of any approaching fronts

Soaring-specific (required for task quality assessment):
- **Thermal indices**: Lifted Index, CAPE, Convective Inhibition (CIN)
- **Boundary Layer (BL) top height** — defines the usable thermal ceiling for the task
- **Thermal strength estimate** — derived from surface heating rate and stability index
- **Top of lift** (convective layer depth) — determines the viable task altitude
- **Orographic effects** — terrain-induced lift and sink along ridge lines (requires elevation/DEM data)
- **Sea breeze front timing** if the task area is within ~60 km of a coastline

### Mesh Optimization

- Apply an initial coarse filter: discard grid points with precipitation > 20% or BL height < 800 m — do not run full soaring analysis on those cells
- Only fetch full soaring indices for grid points that survive the precipitation and stability filter
- Cache weather grid results keyed by `(lat_2dp, lon_2dp, date, model_run)` with a 2-hour TTL; reuse for the same planning area within that window
- When considering wind direction and speed, make the mesh more eliptical, prefer gliding either upwind or downwind for the most of the flight

## Terrain Data

- Integrate an elevation/DEM layer (OpenTopoData or SRTM via free API)
- Required for: terrain clearance calculations per task leg, south-facing slope identification for thermal hotspots, valley convergence zone detection
- Minimum safe altitude per leg = terrain elevation + 300 m AGL (adjustable by safety profile)
- Do not route a task leg over terrain that exceeds the expected task altitude minus the safety margin

---

## Airspace Data

- Source airspace polygons from **OpenAIP** (free tier) for European airspace; fall back to SkyVector for other regions
- Integrate **NOTAM data** via the ICAO NOTAM API for the selected flight date — temporary restrictions can invalidate segments that static airspace data would allow
- Flag any turnpoints or legs that intersect with active NOTAMs on the selected date
- Apply per-pilot airspace buffer: conservative = 2 km lateral buffer, standard = 0.5 km, aggressive = edge only

## Task Optimization — Objective Function

The optimizer must explicitly rank candidate routes by the following criteria (in order):

1. **Distance match**: task distance within ±5% of the user's target
2. **Thermal coverage**: maximize the number of legs that cross grid cells with BL height ≥ 1200 m and positive thermal indices
3. **Wind exposure**: minimize headwind component on the longest leg; prefer downwind or crosswind legs for speed
4. **Terrain clearance**: all legs must clear terrain by the safety margin at expected task altitude
5. **Airspace safety**: no intersections with active restricted/prohibited zones; flag transponder/CTR zones
6. **Landable field proximity**: prefer routes where no point is more than 15 km from a known landable field

**Algorithm**: generate candidate turnpoint sets geometrically (radial sweep or triangulation from the takeoff airport), score each candidate with the above weights, then pass the top 5 scored candidates to the LLM for final selection and narrative justification. Do not pass raw forecast JSON to the LLM — summarize each grid cell into a single-line descriptor first.

## Implementation Instructions

### Weather API Stack

Use the following sources in combination — each serves a different role:

| Source | Env var | Role | Limit |
|---|---|---|---|
| **Open-Meteo** | *(no key)* | Bulk grid forecast — general parameters (wind, temp, precip, cloud base) via single multi-point POST | Free, unlimited |
| **Windy API** | `WINDY_API_KEY` | Soaring-specific indices: thermal forecast, lapse rate, BL height, CAPE — use the Windy Point Forecast API | 1 000 calls/month free |
| **IMGW-PIB** | `IMGW_API_BASE_URL` | High-resolution actuals and short-range forecasts for Poland; supplement Open-Meteo for Polish tasks | Free, no key |
| **OpenWeatherMap** | `OPENWEATHER_API_KEY` | Secondary fallback for areas outside Open-Meteo coverage; one-call API for pressure/front data | Free tier |
| **NOAA** | `NOAA_API_KEY` | Global backup; use only when task area is outside Europe | Free |

**Strategy**:
1. Run Open-Meteo bulk call first for all mesh points (general parameters + coarse BL estimate)
2. Apply precipitation/stability filter — discard cells that fail
3. For surviving cells only, call Windy Point Forecast API to get soaring-specific indices (thermal, lapse rate, BL top)
4. For tasks in Poland, supplement with IMGW actuals for the nearest met stations
5. This ordering minimises Windy call count (the most quota-constrained source)

### AI Token Efficiency

- Summarize each surviving grid cell into a single-line descriptor before sending to the LLM; never pass raw forecast JSON. Example: `"51.2N 18.4E: BL=1650m, thermal=moderate, wind=NW15kt@700hPa, Cu, no rain"`
- Use the AI fallback chain (Gemini → Groq → DeepSeek) defined above
- Only send the top 5 candidate routes to the LLM for final narrative scoring — geometry and numeric scoring is done locally first

### Caching

- Store weather grid results in the existing PostgreSQL DB (`weather_cache` table) keyed by `(lat_2dp, lon_2dp, date, model_run, source)` with a 2-hour TTL
- Reuse cached cells across concurrent users in the same area
- Store airspace snapshots per `(bbox_hash, date)` with a 24-hour TTL

### Other Efficiency Rules

- **Lazy loading**: run the coarse precipitation/BL filter first; only call Windy for cells that pass
- **Mesh shape**: use corridor or ellipse shapes (not full circles) to reduce grid point count by up to 50%

### 1. Glider Polar Data Formats

#### XCSoar — `src/Polar/PolarStore.cpp`

Polars are defined as C++ struct entries. The field order (confirmed from `ToPolarInfo()` / `ToPolarShape()`) is:

```
name, max_gross_kg, max_ballast_liters, v1_kmh, w1_ms, v2_kmh, w2_ms, v3_kmh, w3_ms, wing_area_m2, reference_mass_kg, handicap, empty_mass_kg
```

| Field | Type | Description |
|---|---|---|
| `name` | string | Display name, e.g. `"ASG-29 (15m)"` |
| `max_gross_kg` | int | Max takeoff weight (dry, no ballast) in kg |
| `max_ballast_liters` | int | Maximum water ballast in liters |
| `v1`, `v2`, `v3` | float | Three reference speeds in km/h |
| `w1`, `w2`, `w3` | float | Sink rates at v1/v2/v3 in m/s (negative) |
| `wing_area_m2` | float | Wing area in m², `0.0` if unknown |
| `reference_mass_kg` | float | Mass at which polar was measured; `0.0` = use `max_gross_kg` as reference |
| `handicap` | int | Contest handicap index (typically 50–130) |
| `empty_mass_kg` | int | Approximate empty weight in kg |

Example: `{ "ASG-29 (15m)", 362, 165, 108.8, -0.635, 156.4, -1.182, 211.13, -2.540, 9.20, 0.0, 114, 270 }`

The repo contains **~130 glider types** ranging from paragliders (Para EN A) to 26m motorgliders (Stemme S-12). Notable Polish types included: SZD-51-1 Junior, SZD-55-1 Promyk, SZD-55-2 Diana 2, SZD-50 Puchacz, SZD-48 Jantar, SZD-9bis Bocian, multiple LAK variants.

#### LK8000 — `Common/Distribution/LK8000/_Polars/*.plr`

Text (CSV-ish) format, one glider per `.plr` file. Header comment describes fields:

```
* MassDryGross[kg], MaxWaterBallast[liters], Speed1[km/h], Sink1[m/s], Speed2, Sink2, Speed3, Sink3, WingArea[m2]
```

Example (`ASG29-15.plr`):
```
362,  165,  108.8,  -0.635,  156.4,  -1.182,  211.13,  -2.540, 9.20
```

The first 9 numeric fields of the XCSoar struct **exactly match** the LK8000 PLR format — so a unified import parser can read both sources.

**Import strategy for `glider_polars` PostgreSQL table**:
1. Parse all XCSoar `PolarStore.cpp` entries (primary source, includes handicap + empty mass)
2. Supplement with any LK8000 `.plr` files for gliders not in XCSoar (identify by name comparison)
3. Compute polar coefficients `(a, b, c)` of `w = a*v² + b*v + c` by least-squares fit through the 3 data points — this is needed for MacCready optimisation

### 4. Key Functions to Port (from TypeScript to Python)

#### 4a. `estimate_thermal_index_from_cape()` — `openmeteo.ts`

```python
def estimate_thermal_index_from_cape(
    cape: float,           # J/kg
    temperature: float,    # °C at 2m
    dew_point: float,      # °C at 2m
    cloud_cover_low: float # % 0-100
) -> float:
    """Returns thermal index 0–10."""
    # CAPE component (0–5 pts)
    if cape < 50:    index = 0.0
    elif cape < 200: index = 0.5
    elif cape < 400: index = 1.0
    elif cape < 700: index = 2.0
    elif cape < 1000: index = 3.0
    elif cape < 1500: index = 4.0
    elif cape < 2000: index = 4.5
    else:            index = 5.0

    # Temperature component (0–3 pts)
    if temperature > 25:   index += 3
    elif temperature > 20: index += 2
    elif temperature > 15: index += 1

    # Temp-dew spread component (0–2 pts — indicates convective inhibition)
    spread = temperature - dew_point
    if spread > 12:   index += 2.0
    elif spread > 8:  index += 1.5
    elif spread > 5:  index += 1.0
    elif spread > 3:  index += 0.5

    # Penalty factors
    if cloud_cover_low > 80: index -= 2
    elif cloud_cover_low > 60: index -= 1
    if cape > 2500: index -= 1  # overdevelopment risk

    return round(max(0.0, min(10.0, index)), 1)
```

#### 4b. Cloud base estimation — `openmeteo.ts`

```python
def estimate_cloud_base(
    freezing_level_m: float,   # metres ASL
    temperature: float,        # °C
    dew_point: float           # °C
) -> int:
    """Returns cloud base in feet AGL. Use min of two methods."""
    spread = temperature - dew_point
    convective_base_ft = int(spread * 400)   # classic lifted condensation level formula
    freezing_level_ft  = int(freezing_level_m * 3.28084)
    return max(200, min(convective_base_ft, freezing_level_ft))
```

#### 4c. `calculate_wind_components()` — `aiService.ts`

```python
import math

def calculate_wind_components(
    wind_dir: int,      # degrees true
    wind_speed: float,  # knots
    runway_heading: int # degrees magnetic (use same reference as wind_dir)
) -> dict:
    """Decompose wind into headwind, tailwind, crosswind for a given runway."""
    angle_diff = math.radians(wind_dir - runway_heading)
    headwind  = wind_speed * math.cos(angle_diff)   # positive = headwind
    crosswind = wind_speed * math.sin(angle_diff)   # positive = right crosswind
    return {
        "headwind":  round(headwind, 1),
        "tailwind":  round(-headwind, 1) if headwind < 0 else 0.0,
        "crosswind": round(abs(crosswind), 1),
    }
```

#### 4e. `safe_json_parse()` — `aiService.ts`

```python
import re, json

def safe_json_parse(text: str) -> dict | None:
    """Multi-step JSON repair for LLM responses."""
    # Step 1: try as-is
    try:
        return json.loads(text)
    except json.JSONDecodeError:
        pass

    # Step 2: extract first JSON block
    match = re.search(r'\{[\s\S]*\}', text)
    if match:
        candidate = match.group(0)
        try:
            return json.loads(candidate)
        except json.JSONDecodeError:
            pass

        # Step 3: fix common LLM artifacts
        candidate = re.sub(r',\s*([}\]])', r'\1', candidate)  # trailing commas
        candidate = re.sub(r'[\x00-\x1F\x7F](?=[^"]*")', ' ', candidate)  # control chars
        try:
            return json.loads(candidate)
        except json.JSONDecodeError:
            pass

    return None
```

### 10. Open Questions / Items Requiring Decision Before Implementation

1. **`weather_cache` scope**: the reference project caches per-airport; the task planner needs per-grid-point caching. Table schema above extends this.
2. **Windy API latitude**: the low monthly free limit (1,000 calls) means Windy should only be called for the final surviving mesh points after Open-Meteo coarse filter — confirm this strategy is implemented correctly.
3. **Polar polynomial fit**: the 3-point polar from XCSoar is sufficient for MacCready cruise speed calculation but may not be accurate at very low or very high speeds. Consider whether a simple quadratic fit is acceptable for the MVP.
4. **`reference_mass` vs `max_gross`**: XCSoar uses `0.0` for `reference_mass` when the polar was measured at max_gross. The Python import must handle this by using `max_gross_kg` as the reference when `reference_mass_kg == 0.0`.
5. **NOAA scope**: NOAA is only useful for tasks outside Europe. Since the primary user base is Polish clubs, NOAA can be deprioritised and omitted from the MVP.
6. **Bilingual AI**: the EN+PL single-call optimisation from the reference project is worth replicating exactly — it halves the AI quota usage for Polish deployments.

### 15. AccuracyService — Forecast Verification System (from `accuracyService.ts`)

Tracks forecast accuracy vs. pilot-submitted actual observations. Enables automatic model weight adjustment.

#### Accuracy scoring formula
```python
def calculate_accuracy(forecast: dict, actual: dict) -> dict:
    """Compare a forecast record vs. an actual weather observation."""
    # Wind speed accuracy: ±10 kts = 100% → 0%
    speed_diff = abs(forecast['wind_speed'] - actual['wind_speed'])
    speed_acc = max(0, 100 - (speed_diff / 10) * 100)

    # Wind direction accuracy: circular ±30° = 100% → 0%
    dir_diff = ((forecast['wind_direction'] - actual['wind_direction'] + 180) % 360) - 180
    dir_acc = max(0, 100 - (abs(dir_diff) / 30) * 100)

    wind_acc = speed_acc * 0.7 + dir_acc * 0.3   # weighted

    # Temperature accuracy: ±5°C = 100% → 0%
    temp_acc = max(0, 100 - (abs(forecast['temperature'] - actual['temperature']) / 5) * 100)

    # Cloud base accuracy (if both present): ±2000 ft = 100% → 0%
    cloud_acc = None
    if forecast.get('cloud_base') and actual.get('cloud_base'):
        cloud_acc = max(0, 100 - (abs(forecast['cloud_base'] - actual['cloud_base']) / 2000) * 100)

    # Overall weighted: wind 50%, temp 30%, cloud base 20% (renormalized if cloud missing)
    weights = {'wind': 0.5, 'temp': 0.3, 'cloud': 0.2 if cloud_acc is not None else 0}
    total_w = sum(weights.values())
    overall = (wind_acc * weights['wind'] + temp_acc * weights['temp'] + (cloud_acc or 0) * weights['cloud']) / total_w
    return {'wind_accuracy': wind_acc, 'temperature_accuracy': temp_acc,
            'cloud_base_accuracy': cloud_acc, 'overall_accuracy': round(overall, 2)}
```

#### Model weight update formula
```
accuracy% → model weight in [0.0, 1.5]:
  ≥ 40%: weight = clamp( (accuracy - 60) / 40 + 0.5, 0.0, 1.5 )
  < 40%: weight = 0.0 (disabled)
  90%  → 1.25; 80% → 1.0 (baseline); 60% → 0.5; 40% → 0.0
```

The weight is stored in `forecast_models.weight` and used to order/rank model results.

## §19 — Glider Polar Database

### Overview

Polar data has been collected from two authoritative sources:

1. **XCSoar `PolarStore.cpp`** — 155 glider entries with full 13-field structs (mass, ballast, 3 speed/sink points, wing area, reference mass, handicap, empty mass)
2. **LK8000 `_Polars/`** — 156 `.plr` files; mostly overlapping with XCSoar but with additional variant entries (pilot-only / passenger-included mass splits, motorglider PAS/PIL variants, flap-equipped gliders, paragliders, hang gliders)

### Data Formats

**XCSoar struct fields (13 per glider):**
```
name | max_gross_kg | max_ballast_l | v1_kmh | w1_ms | v2_kmh | w2_ms | v3_kmh | w3_ms | wing_area_m2 | ref_mass_kg | handicap | empty_mass_kg
```

**LK8000 `.plr` fields (9 per glider):**
```
gross_mass_kg | max_ballast_l | v1_kmh | w1_ms | v2_kmh | w2_ms | v3_kmh | w3_ms | wing_area_m2
```
- Lines starting with `*` are comments (skip)
- Optional 2nd line: flap settings — `ref_mass, count, speed1, flap_id1, speed2, flap_id2, ...`
- `gross_mass_kg` = total all-up weight at polar measurement time (includes pilot weight)
- LK8000 PAS variants = passenger-included mass; PIL variants = pilot-only mass

### Naming Conventions

| LK8000 suffix | Meaning |
|---|---|
| `_PAS` | Passenger (both seats occupied) — higher gross mass |
| `_PIL` | Pilot only (one seat) — lower gross mass |
| `_Wnglts` | Winglets fitted variant |
| `_M` suffix | Motorglider variant |

### Implementation Notes

**Database table design (future `glider_polars` table):**

```sql
CREATE TABLE glider_polars (
    id              SERIAL PRIMARY KEY,
    name            VARCHAR(80) NOT NULL,
    source          VARCHAR(20) NOT NULL,       -- 'xcsoar' | 'lk8000'
    variant         VARCHAR(20),                -- 'PAS' | 'PIL' | 'winglets' | NULL
    max_gross_kg    NUMERIC(7,2) NOT NULL,
    max_ballast_l   NUMERIC(6,1) NOT NULL,
    v1_kmh          NUMERIC(6,2) NOT NULL,
    w1_ms           NUMERIC(6,4) NOT NULL,
    v2_kmh          NUMERIC(6,2) NOT NULL,
    w2_ms           NUMERIC(6,4) NOT NULL,
    v3_kmh          NUMERIC(6,2) NOT NULL,
    w3_ms           NUMERIC(6,4) NOT NULL,
    wing_area_m2    NUMERIC(6,2),
    ref_mass_kg     NUMERIC(6,2),               -- XCSoar only
    handicap        SMALLINT,                   -- XCSoar only
    empty_mass_kg   NUMERIC(6,2),               -- XCSoar only
    has_flaps       BOOLEAN DEFAULT FALSE,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
```

**Polar curve computation** — to convert 3-point (v, w) data to a parabolic glide polar:
- Fit `w = a·v² + b·v + c` using the 3 speed/sink pairs
- Best L/D = max of `−v / w` along the curve
- Min sink speed = `v` at `dw/dv = 0` → `v_ms = −b / (2a)`

**Wing loading correction** — when pilot weight differs from polar reference mass:
- `w_corrected(v) = w_ref(v · √(m_actual/m_ref))` scaled by `√(m_actual/m_ref)`
- LK8000 `WeightOffset` function implements this scaling

**Source priority for UI glider selector:**
1. XCSoar entry (has additional fields: handicap, empty mass, ref mass)
2. LK8000 PAS/PIL split (for two-seaters where user selects seating configuration)
3. LK8000-unique entries (Arcus M, motorglider variants, paragliders, hang gliders)

### Revised `glider_polars` Schema

The §19 SQL schema and the §2 schema both define `glider_polars` but with slight differences. Here is the **canonical merged schema** that matches both the XCSoar + LK8000 data and our UI needs:

```sql
CREATE TABLE IF NOT EXISTS glider_polars (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            VARCHAR(200) NOT NULL,
  source          VARCHAR(20) NOT NULL,           -- 'xcsoar' | 'lk8000' | 'manual'
  variant         VARCHAR(20),                    -- 'PAS' | 'PIL' | 'winglets' | NULL
  max_gross_kg    NUMERIC(7,2) NOT NULL,
  max_ballast_l   NUMERIC(6,1) NOT NULL DEFAULT 0,
  v1_kmh          DOUBLE PRECISION NOT NULL,
  w1_ms           DOUBLE PRECISION NOT NULL,
  v2_kmh          DOUBLE PRECISION NOT NULL,
  w2_ms           DOUBLE PRECISION NOT NULL,
  v3_kmh          DOUBLE PRECISION NOT NULL,
  w3_ms           DOUBLE PRECISION NOT NULL,
  wing_area_m2    DOUBLE PRECISION,
  reference_mass_kg DOUBLE PRECISION,             -- mass at which polar was measured
  handicap        SMALLINT,                       -- contest handicap index
  empty_mass_kg   NUMERIC(6,2),
  has_flaps       BOOLEAN DEFAULT FALSE,
  -- Derived polynomial: w = a*v² + b*v + c  (v in m/s, w in m/s)
  polar_a         DOUBLE PRECISION,
  polar_b         DOUBLE PRECISION,
  polar_c         DOUBLE PRECISION,
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (name, source, variant)
);
CREATE INDEX ON glider_polars (name);
CREATE INDEX ON glider_polars (source);
```

**Key design decisions:**
- `UNIQUE (name, source, variant)` instead of `UNIQUE (name)` — allows same glider from both sources and PAS/PIL variants
- `variant` column — captures PAS/PIL/winglets without name mangling
- `polar_a/b/c` — pre-computed on insert so the frontend/service never needs to solve the quadratic
- `NUMERIC(7,2)` for mass fields — handles values like 472.50 (ICP Savannah has 472.5)
- No `SERIAL` PK — uses UUID consistent with the rest of the schema from §2

### Migration Plan: Seed Script

The seed script will:
1. Parse all 155 XCSoar entries from the table in §19
2. Parse all 62 LK8000-only entries from the supplement table in §19
3. For each entry, compute `polar_a`, `polar_b`, `polar_c` by solving the 3-point system
4. Insert with `ON CONFLICT (name, source, variant) DO UPDATE` for idempotent re-runs

```python
# Polar coefficient computation (convert km/h → m/s first)
import numpy as np

def compute_polar_coefficients(v1_kmh, w1, v2_kmh, w2, v3_kmh, w3):
    """Fit w = a*v² + b*v + c through 3 speed/sink points.
    Speeds converted from km/h to m/s for SI coefficients."""
    v1 = v1_kmh / 3.6
    v2 = v2_kmh / 3.6
    v3 = v3_kmh / 3.6
    A = np.array([
        [v1**2, v1, 1],
        [v2**2, v2, 1],
        [v3**2, v3, 1],
    ])
    b_vec = np.array([w1, w2, w3])
    a, b, c = np.linalg.solve(A, b_vec)
    return a, b, c
```
